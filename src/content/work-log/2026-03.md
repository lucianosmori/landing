March was about catching up on three things I had been pushing off.

## Public CAs, private CAs, and the mid-2026 cutover

By June 2026, Chrome's root program will require that any certificate hierarchy in its public trust store be dedicated to TLS server authentication only. Public certificate authorities will no longer be allowed to issue certs that carry both `id-kp-serverAuth` and `id-kp-clientAuth` Extended Key Usages. The two roles are being split into separate PKIs.

Worth being precise about why this matters. A TLS certificate's EKU field declares what role the cert is allowed to play. `serverAuth` covers the case most people associate with HTTPS, where the cert is presented by a server proving its identity to a connecting client. `clientAuth` covers the inverse: the cert is presented by the connecting peer to prove its own identity. That second one shows up everywhere off the public web — VPN onboarding, Wi-Fi 802.1X, device-to-cloud auth, mutual TLS between microservices, single sign-on flows that ship a cert to the endpoint, DevOps workload identity. Anywhere a cert is presented by something that is not a public web server.

A lot of organizations have been issuing public certs and quietly using them for those internal client-auth scenarios. It worked, partly because public CAs were free and automated and partly because nothing forced the question. The mid-2026 cutover forces it. Public CAs were never designed for internal authentication workflows. They are subject to external audits, browser-driven policy changes, and constraints on what they are allowed to issue. They are tuned for the public web, not for the flexibility and control that internal authentication actually needs.

The path forward is a private certificate authority, run by you, with certificate profiles and policies tuned to the internal use cases you actually have. You get control over issuance, revocation, EKU sets, SAN shapes, and renewal cadence. You can integrate with ACME, EST, or SCEP for automation, and you remove a public-trust-store dependency from your internal flows entirely. The setup work is real (HSM-backed key material, a signing endpoint that is auditable and rate-limited, automated renewal that does not page someone at 2am), but every part of it is the kind of decision a senior platform team should be able to make on a whiteboard.

The deeper lesson underneath the deadline: SSL/TLS, X.509 fields, EKU semantics, key hygiene, and PKI in general are not niche specialties. They are the substrate that almost every other security control sits on top of. A team fluent in the difference between a CA, an intermediate, an EKU, and a SAN, with the muscle memory to read `openssl x509 -text` output on a bad day, navigates the mid-2026 transition without drama. A team that is not, hits the deadline as a surprise.

## A real agentic engineering setup

Running several coding agents on the same repository at the same time is doable, but only if each one has its own filesystem. Process-level isolation is not enough; in-place edits from two agents collide on the working tree the first time they both touch the same file. Worktrees turn out to be the right unit of isolation, because they map directly onto the agent's mental model of "the repo." I settled on a `<repo>-wt-<task>` naming convention, with one branch per worktree and a clean teardown when the task ends.

Adjacent change that paid off immediately: per-project MCP scoping. The default MCP loadout was schema-noisy enough to evict useful context on every turn — every tool definition has a JSON schema, and a pile of tools you never use still sits in the prompt. Trimming the loadout to the servers each project actually needs cut the per-turn schema overhead noticeably and freed budget for the work itself.

Treating multi-agent code review as its own discipline started paying off too. Different agents have different default biases. One leans hard on type-system invariants and barely notices concurrency bugs; another flags everything that vaguely resembles a race condition and skips most style issues. Without a way to weigh those voices, the noise compounds and people stop reading the comments. The fix is a paired reviewer-fingerprint file per agent: what it reliably catches, and the categories where it tends to false-positive. With that file in hand, each agent's review becomes a recognizable voice you can read at a glance instead of a wall of text.

The biggest anti-pattern I had to write a hard rule against this month was agents opening PRs without explicit confirmation. The fix is not a per-task reminder — those evaporate the next session. It is a sentence in the agent's persistent instructions, alongside the other don't-touch-prod rules. Once that lives at the top, the per-task prompt gets shorter and the surprise factor drops to zero.

Parallel thread, less mature: experimenting with fleet-style background workers (agents that run on a schedule, sweep stale issues, propose dependency bumps, watch for known-bad patterns in new code) versus the always-interactive harness on a laptop. The two have different failure modes — interactive agents fail loudly in front of you, background ones fail silently in a log somewhere — and want different MCP tooling and different oversight. More on this once it has settled into a usable shape.

## Infra at scale

A decade ago, a few hundred config values in a shared store felt like a lot. Today, thousands across multiple regions is normal. The per-account, per-region throttle that nobody notices at small scale becomes a self-inflicted denial of service the first time a burst of workloads cold-start at the same time and all reach for the same store. The lesson is the same one as last year and the year before: capacity planning is mostly about what gets called at the worst possible moment, not what gets called on the happy path. Throttles, rate limits, and burst quotas hide in plain sight until traffic actually shows up.

The other thing that becomes obvious at scale is the cost of letting state accumulate without a sweep policy. An audit of an established parameter store will turn up orphans almost immediately: leftovers from deleted environments, stale credentials from rotated services, doubled-up entries from a migration that finished six months ago and was never cleaned up. None of it costs much per item, but the human cost of "what is this and is anyone reading it" goes up with every entry, and the security cost of leaving a stale credential in a place a workload can still read it is real. A scheduled discovery pass that flags anything not referenced in IaC is the cheapest preventative.

The same shape applies to resource tagging. A mandatory tag schema (Owner, Environment, CostCenter, Criticality, DataSensitivity, and a small number of others) only works when validation lives at multiple layers: variable-level checks at IaC plan time, advisory tag policies at the cloud-account level, and continuous compliance scans for anything that drifts when somebody touches a resource in the console. Defense in depth, because each layer catches different failure modes, and cost-allocation reports, incident routing, and lifecycle automation all degrade silently when the schema slips.

Last thread of the month: splitting deploy roles per environment so a misconfigured pipeline cannot accidentally apply to production. Trust policies scoped to a specific repository and ideally a specific branch. Long-lived access keys in CI secrets are obsolete; OIDC-based token exchange is the boring, correct default. The day-one cost is real (one role per environment, one trust policy per role, one set of permissions to right-size), but the day-two and day-N savings are larger than they look until you have lived through the alternative.
